# v2 引擎研发失败复盘报告

## 1. 结论

这份报告的核心结论很明确：

1. 当前 `v2` 引擎在编配效果上已经失败，不能继续作为主引擎路线推进。
2. `v1` 并不是“老旧但勉强能用”的过渡方案，而是在当前任务、当前数据条件、当前工程成熟度下明显更成功的产品级基线。
3. `v2` 的问题不是若干局部 bug 的偶发叠加，而是整体架构假设、主决策器能力、时间粒度设计和目标函数定义同时失配。
4. 后续正确路线不是继续把 `v2` 当主链路小修小补，而是恢复 `v1` 为主基线，把 `v2` 降级为研究分支或局部模块实验场。

换句话说，`v2` 的失败不是“实现还差一点”，而是“当前这条架构路线在现阶段不成立”。

## 3. 事实基线

### 3.1 从结果看

在《让我们荡起双桨》样例上：

1. `v1` 生成的和声整体落在 `Eb / Ab / Bb / Cm` 语境内，终止感、情绪色彩、风格适配都接近人工可接受水平。
2. 初始版 `v2` 出现了 `F / G / C / G#m` 这样的错误和弦，属于严重乐理失真。
3. 在修复判调与拼写后，`v2` 虽然回到调内，但仍一度出现“Ab 开头、Cm 收尾、Cm 过密”的问题，说明问题不止是判调错误。

这说明：

1. `v2` 的失败不是单点故障。
2. 即使修正最明显的错误，整体审美与结构问题仍然存在。
3. `v1` 的优势不是偶然，而是其主决策体系更接近“成熟编配”的生成方式。

### 3.2 从代码结构看

`v1` 实际主链路是：

`MusicXML -> 解析 -> 调性分析 -> 旋律特征 -> RAG -> LLM -> 难度过滤 -> 验证`

对应实现见：
[engine_version_history/enginev1/harmonize-pipeline.ts](/c:/Users/卢易航/Desktop/Web应用/和声分析引擎/engine_version_history/enginev1/harmonize-pipeline.ts)

`v2` 实际主链路是：

`MusicXML -> 分析层 -> 候选格 -> 全局解码 -> 三层修复 -> 置信度 -> 输出`

对应实现见：
[harmony-engine/src/harmonizer/harmony-engine-pipeline.ts](/c:/Users/卢易航/Desktop/Web应用/和声分析引擎/harmony-engine/src/harmonizer/harmony-engine-pipeline.ts)

设计上，`v2` 更宏大、更模块化；结果上，`v2` 明显更差。

这意味着问题不能只用“v2 还没调完”来解释，必须上升到架构哲学层面复盘。

## 4. 为什么 v1 在效果上成功

### 4.1 v1 的主决策器是真正强先验模型

`v1` 的核心智能不是一堆手工规则，而是：

1. 由旋律特征构造 prompt
2. 用 RAG 提供真实歌曲片段作为参考
3. 让 LLM 直接输出整段和声

相关实现：
[harmony-engine/src/harmonizer/llm-harmonizer.ts](/c:/Users/卢易航/Desktop/Web应用/和声分析引擎/harmony-engine/src/harmonizer/llm-harmonizer.ts)
[harmony-engine/src/harmonizer/melody-features.ts](/c:/Users/卢易航/Desktop/Web应用/和声分析引擎/harmony-engine/src/harmonizer/melody-features.ts)

这意味着 `v1` 在生成和声时，天然具备以下综合能力：

1. 旋律整体语义判断
2. 常见流行/儿歌伴奏模式记忆
3. 情绪色彩与风格的一体化判断
4. 终止感与段落感的整体把控
5. 局部与整体平衡

这些能力没有被显式拆成模块，但它们在大模型中是被联合编码的。

### 4.2 v1 的输出目标是“像人写的伴奏”

`v1` 的目标很直接：

“为这段旋律生成一版正常、顺耳、可弹、像样的和声。”

这是一个听觉导向、结果导向的问题定义。

它不是先问：

1. 这个和弦覆盖了几个旋律音？
2. 这个低音进行有多平滑？
3. 这个时间窗属于哪个功能状态？

而是直接让主决策器去优化整体听感。

### 4.3 v1 的约束是后置收敛，不是前置束缚

`v1` 先生成，再通过难度白名单收紧输出：
[engine_version_history/enginev1/difficulty-filter.ts](/c:/Users/卢易航/Desktop/Web应用/和声分析引擎/engine_version_history/enginev1/difficulty-filter.ts)

这个顺序非常关键。

它的效果是：

1. 先让模型给出整体像样的方案
2. 再把过于复杂、过于花哨的和弦压回教学或常规伴奏可接受范围

所以 `v1` 输出往往“稳、顺、保守但正确”。

对于儿歌、教学和常见歌曲伴奏，这恰好是对的。

### 4.4 v1 的判调器虽然不完美，但至少和谱面信息一致

`v1` 的调性分析不是完全盲判，它会把算法结果与 MusicXML 调号对照，并在相对大小调歧义时优先信任谱面信息：
[harmony-engine/src/analyzer/key-analyzer.ts](/c:/Users/卢易航/Desktop/Web应用/和声分析引擎/harmony-engine/src/analyzer/key-analyzer.ts)

这使得 `v1` 至少站在正确调性上思考。

在和声任务里，这一点是底线，不是加分项。

## 5. 为什么 v2 在效果上失败

## 5.1 v2 名义上是“三路候选 + 全局搜索”，实际上主要靠规则在跑

`v2` 的设计想法是：

1. 规则路由
2. 检索路由
3. 模型路由

一起生成候选，再全局解码。

但在当前实现里：

1. 检索路由默认没有 provider，会直接返回空：
   [retrieval-router.ts:28](/c:/Users/卢易航/Desktop/Web应用/和声分析引擎/harmony-engine/src/candidate/retrieval-router.ts#L28)
2. 模型路由默认也没有真正的 symbolic model 或 LLM provider，只会退回保守候选：
   [model-router.ts:42](/c:/Users/卢易航/Desktop/Web应用/和声分析引擎/harmony-engine/src/candidate/model-router.ts#L42)

因此，`v2` 实际上并不是“学习型候选系统”，而是“规则候选系统 + 手工打分系统”。

这和设计目标已经发生根本偏离。

### 5.2 v2 的主优化目标不是听感，而是代理指标

`v2` 的解码器主要依赖这些局部指标：

1. melody coverage
2. beat alignment
3. function progression
4. bass smoothness
5. style transition
6. key continuity

见：
[global-decoder.ts](/c:/Users/卢易航/Desktop/Web应用/和声分析引擎/harmony-engine/src/decoder/global-decoder.ts)

这些指标在理论上有意义，但它们不是“好听”的充分条件。

典型错误是：

1. 某个和弦局部覆盖旋律很高
2. 于是它在局部得分上占优
3. 但它在整首曲子的情绪与句法上完全不合适

例如 `Cm` 在 `Eb major` 里局部经常“算得通”，但如果使用过密，就会把阳光大调歌曲推向关系小调色彩。

### 5.3 v2 的时间粒度是错位的

这是一个架构级问题。

在 `v2` 中：

1. `key_sequence` 按窗口生成
2. `functional_states` 基于 `key_sequence` 计算
3. `time_spans` 则由节奏预测器另行生成
4. 候选格又基于 `time_spans` 展开

相关实现：
[harmony-engine/src/analyzer/key-sequence-analyzer.ts](/c:/Users/卢易航/Desktop/Web应用/和声分析引擎/harmony-engine/src/analyzer/key-sequence-analyzer.ts)
[harmony-engine/src/harmonizer/functional-state-tracker.ts](/c:/Users/卢易航/Desktop/Web应用/和声分析引擎/harmony-engine/src/harmonizer/functional-state-tracker.ts)
[harmony-engine/src/analyzer/harmonic-rhythm-predictor.ts](/c:/Users/卢易航/Desktop/Web应用/和声分析引擎/harmony-engine/src/analyzer/harmonic-rhythm-predictor.ts)

这些模块各自看都合理，但它们不共享严格一致的时间语义。

结果是：

1. 分析层以为自己在约束候选层
2. 实际上很多约束只是索引对齐，不是音乐时间上的严格对齐
3. 于是“功能状态”“句法边界”“节奏跨度”经常在不同模块中说的不是同一个东西

这会直接削弱全局约束的有效性。

### 5.4 v2 的 repair 层会推翻全局决策

`v2` 不是只有一个 decoder 在做决策，后面还有三层 repair：
[three-layer-repairer.ts](/c:/Users/卢易航/Desktop/Web应用/和声分析引擎/harmony-engine/src/repair/three-layer-repairer.ts)

这导致一个很危险的现象：

1. 全局解码器刚选出相对合理的起始主和弦
2. repair 层又因为局部覆盖率更高，把它换成了局部更“对音”的和弦
3. 全局结构再次被局部规则打坏

换句话说，`v2` 的系统内部没有一个真正统一的裁决中心。

这也是为什么它会出现：

1. 开头漂浮
2. 句尾不稳
3. 终曲收束失真

这些都不是单个模块的问题，而是“谁说了算”没有被架构统一。

### 5.5 v2 的风格控制只是表格式偏置，不是整体风格理解

`v2` 的风格系统本质是：

1. preferred chords
2. transition matrix
3. extension allowance

见：
[style-control-system.ts](/c:/Users/卢易航/Desktop/Web应用/和声分析引擎/harmony-engine/src/harmonizer/style-control-system.ts)

这最多算“风格微调器”，不是“风格理解器”。

它能帮助系统在若干候选间微调倾向，但不能像 `v1` 的 LLM 那样整体理解：

1. 这是一首明亮儿歌
2. 不该频繁染上关系小调色彩
3. 结尾必须回主和弦
4. 第一小节最好站稳调性中心

所以 `v2` 即使“没有离调”，仍然会“情绪失真”。

## 6. 最根本的架构差异

把问题说透，真正的差异不是“LLM vs 搜索”，而是下面这三点。

### 6.1 v1 借用了现成的音乐脑，v2 试图自造音乐脑

`v1` 的主决策器是外部强模型，它已经学会了大量和声常识。  
`v2` 想把音乐理解拆成若干模块和手工权重，再通过搜索重新组合。

问题是：

1. `v2` 没有受过训练的 learned scorer
2. 没有大规模标注曲库上的参数拟合
3. 没有证明这些代理指标能逼近真实听感

所以它本质上是在“没有大脑的情况下模拟大脑”。

### 6.2 v1 的目标函数更接近用户价值，v2 的目标函数更接近工程可解释性

用户真正要的是：

1. 顺耳
2. 像样
3. 风格对
4. 可以弹

`v1` 在做这件事。  
`v2` 更像在做：

1. 指标可拆
2. 打分可解释
3. 模块化可控

这些在工程上有价值，但如果它们无法稳定导出更好的结果，就不能代替真正的音乐效果。

### 6.3 v1 的失败方式是“偶尔不稳”，v2 的失败方式是“系统性失真”

`v1` 的主要风险是：

1. 偶尔风格漂移
2. 个别段落生成不稳定
3. 某些结果不够可解释

但它多数时候仍然像音乐。

`v2` 的风险则是：

1. 判调错就整条链错
2. 候选空间错就搜索无解
3. 打分错就系统性偏向错误和弦
4. repair 再次把全局结构破坏

也就是说，`v2` 的失败是整套系统逻辑一致地把结果推向错误方向。

这比 `v1` 的波动更致命。

## 7. 本次复盘后的战略判断

### 7.1 v1 应恢复为正式基线

从现在开始，应明确把 `v1` 定义为：

1. 当前主引擎
2. 效果基准
3. 回归标准

不是“临时保底”，而是“正式 baseline”。

### 7.2 v2 不应继续作为主链路

`v2` 目前不具备以下资格：

1. 不具备效果优于 `v1` 的证据
2. 不具备稳定性优于 `v1` 的证据
3. 不具备大规模测试优于 `v1` 的证据

因此 `v2` 不应继续承担主引擎职责。

### 7.3 v2 只能以“局部替换实验”的形式存活

如果 `v2` 还有价值，它的价值应变成：

1. 交互编辑层
2. 候选解释层
3. 置信度层
4. 局部修复层
5. 评测与分析工具链

也就是说，`v2` 适合做“外挂模块”，不适合做“主决策器”。

## 8. 对 v3 的建议

如果未来要做 `v3`，正确路线不是继续沿着当前 `v2` 整体架构硬推，而是：

### 8.1 原则一：保留 v1 主生成，逐模块替换

推荐路线：

1. 以 `v1` 为主生成引擎
2. 用 `v2` 的模块做旁路分析
3. 逐步证明单模块替换能稳定提升效果
4. 只有在基准集稳定优于 `v1` 时，才接管主链路

### 8.2 原则二：先建立评测体系，再谈新架构

必须建立固定 benchmark：

1. 儿歌
2. 流行歌
3. 赞美诗
4. 教学旋律
5. 有转调样例

每次替换模块时都要回答：

1. 效果是否优于 `v1`
2. 是否减少人工修改率
3. 是否提升终止正确率
4. 是否减少风格跑偏

没有这套评测，任何“更先进架构”都只是主观感觉。

### 8.3 原则三：如果继续做搜索式引擎，必须补上 learned scorer

如果未来仍想走“候选格 + 搜索”路线，必须补足：

1. 训练得到的评分器
2. 基于人工标注或用户修正数据的参数拟合
3. 真实听感目标的监督信号

否则搜索式引擎永远只是在弱候选和手工权重之间兜圈子。

## 9. 最终结论

这次 `v2` 研发的失败，反而暴露出一个更重要的事实：

`v1` 并不是“原始版本”，而是当前系统里唯一真正成功地把音乐效果、工程可用性和产品价值统一起来的核心引擎。

`v2` 的失败说明：

1. 模块化、可解释、可控，并不自动等于更好的音乐结果
2. 在没有真正强主决策器的情况下，拆模块和堆打分只会制造复杂性
3. 音乐生成系统最重要的不是架构图好看，而是主决策器是否真的会写音乐

因此，当前阶段最正确的战略不是“继续挽救 `v2`”，而是：

1. 恢复 `v1` 为主引擎
2. 以 `v1` 为基线做效果评测
3. 把 `v2` 重构为旁路能力和研究模块
4. 未来如果要做 `v3`，必须走“基于 `v1` 逐模块替换”的演进路线，而不是再来一次整体替代

这不是保守，而是基于实际效果的工程理性。
