一、项目概述
────────────────────────────────────────────────────────────────────────────────

MeloChord 是一个基于 RAG（检索增强生成）+ LLM（大语言模型）的自动和声分析
平台。用户上传旋律（MusicXML 文件或乐谱图片/PDF），系统自动为其配置和弦，生成
带和弦标注的领谱（Lead Sheet），并提供五线谱和简谱双视图可视化。

核心用户场景：辅助音乐老师即兴伴奏、辅助编曲学习者理解和声进行。

技术栈：TypeScript + Node.js + Express，前端为单页 HTML 应用，无框架依赖。


二、系统架构
────────────────────────────────────────────────────────────────────────────────

完整处理流程：

  乐谱图片/PDF ──→ OMR 识别（Audiveris）──→ MusicXML
                                                │
  MusicXML 文件 ────────────────────────────────┘
        │
        ▼
  MusicXML 解析器 ──→ 内部表示 IR（Score）
        │
        ├──→ 分段调性分析（MusicXML 调号 + Krumhansl-Schmuckler 验证 + 转调检测）
        │
        ├──→ 旋律特征提取（音程序列 + 节奏密度 + 音级序列）
        │         │
        │         ▼
        │    RAG 检索（text-embedding-v4 向量化 → Hooktheory 知识库余弦相似度匹配）
        │         │
        │         ▼
        │    LLM 和弦生成（Qwen-plus，DashScope API，分块 4 小节处理）
        │         │
        │         ▼
        │    难度白名单过滤（basic/intermediate/advanced 三级）
        │         │
        │         ▼
        │    后处理验证（和弦音覆盖检查 + 转换概率检查）
        │         │
        │         ▼
        │    RNA 和弦 → ChordSymbol 转换（罗马数字级数 → 具体和弦名）
        │
        ▼
  和弦注入 MusicXML（<harmony> 元素）
        │
        ├──→ OSMD 五线谱渲染
        ├──→ Canvas 简谱渲染
        ├──→ ABC Notation 导出
        └──→ Verovio → PNG/PDF 导出


三、模块详细说明
────────────────────────────────────────────────────────────────────────────────

3.1 OMR 光学乐谱识别（src/omr/）

  - audiveris-omr.ts：集成 Audiveris 5.9.0 CLI 进行五线谱识别
    · 支持格式：PNG、JPG、BMP、TIFF、PDF
    · 自动预处理：低分辨率图片（<2400px 宽）使用 sharp 库进行 lanczos3 放大
      并设置 300 DPI 元数据，确保 Audiveris 识别精度
    · 支持文件路径输入（recognizeScore）和内存 Buffer 输入（recognizeBuffer）
    · 自动检测 Audiveris 安装路径，支持环境变量 AUDIVERIS_PATH 配置
    · 超时控制，默认 120 秒

  - mxl-unzip.ts：MXL 文件解压器
    · Audiveris 输出 .mxl 格式（压缩的 MusicXML）
    · 纯 TypeScript 实现的 ZIP 解压，通过 Central Directory 解析
    · 支持 Deflate 解压（使用 Node.js zlib）
    · 处理了 data descriptor 场景（本地文件头 compressedSize=0 的情况）

3.2 MusicXML 解析器（src/parser/）

  - musicxml-parser.ts：将 MusicXML 文档解析为内部 Score 表示
    · 使用 fast-xml-parser 进行 XML 解析
    · 提取元信息：曲名、作曲者、调性、拍号、速度
    · 解析音符：音高（音名+变音记号+八度）、时值、附点、连音线
    · 解析休止符
    · 处理多声部钢琴谱：只取第一个 part（右手/主旋律）
    · 和弦音处理：同一拍多个音同时发声时，保留最高音作为旋律音
    · 支持小节内调性变化和拍号变化

  - musicxml-merge.ts：多页 MusicXML 合并工具
    · 将多个 MusicXML 文档（来自多页 OMR 识别）合并为一个完整乐谱
    · 以第一页元信息为基准，后续页小节追加，编号自动递增
    · 保留各页的调性/拍号变化

3.3 核心数据模型（src/core/）

  - types.ts：内部音乐表示 IR（Intermediate Representation）
    · Pitch：音高（音名 + 变音记号 + 八度），保留音名语义（区分 C# 和 Db）
    · Note / Rest：音符和休止符，含时值、附点、连音线、拍位置
    · ChordSymbol：和弦标记（根音 + 质量 + 拍位置）
    · Measure：小节（音乐事件列表 + 和弦标注 + 调性/拍号变化）
    · Score：完整乐谱（曲名、作曲者、调性、拍号、速度、所有小节）
    · 支持 11 种和弦质量：大/小/减/增三和弦，属七/大七/小七/减七/半减七，sus2/sus4
    · 支持 6 种时值：全/二分/四分/八分/十六分/三十二分音符

  - constants.ts：乐理常量
    · 音名到半音映射、变音记号偏移
    · 五度圈调性映射（fifths -7 到 +7）
    · 大调/小调音阶音程模板
    · 和弦音程模板（11 种和弦质量）
    · MusicXML 时值映射

3.4 调性分析器（src/analyzer/）

  - key-analyzer.ts：双路径调性分析 + 转调检测
    · 主路径：直接读取 MusicXML 中的调号信息
    · 验证路径：Krumhansl-Schmuckler 算法
      - 统计乐谱中所有音符的加权音高分布（时值作为权重）
      - 与 Krumhansl-Kessler 大调/小调音高分布模板计算皮尔逊相关系数
      - 遍历所有 24 个调（12 大调 + 12 小调），找到最佳匹配
    · 智能决策：
      - MusicXML 与 KS 一致 → 高置信度采用
      - 关系大小调差异 → 根据相关系数差距决定
      - 完全不一致 → KS 相关系数 >0.8 时采用 KS 结果
    · 分段调性分析（analyzeKeySegmented）：
      - 滑动窗口检测转调点
      - 输出初始调性 + 转调列表（小节号、新调、置信度）
    · 输出：调性 + 置信度 + 分析来源 + top-5 候选调性

3.5 和声引擎（src/harmonizer/）

  - melody-features.ts：旋律特征提取
    · 从 Score 提取：音程序列、节奏模式、音级序列（相对于主音）
    · 计算节奏密度（每拍平均音符数）
    · 生成 RAG 检索查询文本（每 2 小节一个查询）
    · 生成 LLM prompt 描述文本

  - rag-retriever.ts：RAG 检索器
    · 使用阿里云 DashScope text-embedding-v4 模型进行文本向量化
    · 知识库：Hooktheory 数据集（125,837 个有效片段，来自 26,175 首歌曲）
    · 三级加载策略（按优先级）：
      1. 二进制格式：phrase_meta.json（20MB）+ phrase_embeddings.bin（492MB Float32Array）
         → 加载快，内存友好，检索时零额外 API 调用
      2. JSON 格式：phrase_embeddings.json（2.7GB）
         → 可能因 V8 字符串长度限制失败
      3. 无 embedding 回退：hooktheory_phrases.json
         → 检索时实时 embed 候选片段，极慢（每 chunk ~50s）
    · 预筛选：按调式过滤候选
    · 有预计算 embedding 时：全量候选参与余弦相似度排序
    · 无预计算 embedding 时：按旋律长度粗筛到 200 条，实时 embed
    · cosineSimilarityWithBuffer：直接在 Float32Array 上计算，零中间数组分配
    · 返回 top-K 最相似片段（默认 K=5）

  - llm-harmonizer.ts：LLM 和弦生成器
    · 使用阿里云 DashScope Qwen-plus 模型
    · 系统 prompt 包含和声规则和难度级别指导
    · 用户 prompt 包含旋律特征描述 + RAG 检索到的参考和弦进行
    · 三级难度：
      - basic：只用 I, IV, V, vi 四个基本和弦
      - intermediate：顺阶七和弦 + 常见替代和弦
      - advanced：离调和弦、借用和弦、挂留和弦等
    · temperature=0 保证输出确定性
    · RNA（罗马数字级数）→ ChordSymbol 转换：
      支持升降号前缀、大小写区分大小和弦、多种质量后缀（7, maj7, dim, aug 等）
    · 输出解析：正则匹配 "小节号: 和弦1 和弦2" 格式

  - difficulty-filter.ts：难度白名单过滤
    · 根据难度级别对 LLM 生成的 RNA 和弦进行白名单过滤
    · 不在白名单中的和弦替换为功能最接近的允许和弦
    · 功能替换映射：按主功能/下属功能/属功能组归类
    · advanced 级别不做任何过滤

  - chord-validator.ts：后处理验证层
    · 和弦音覆盖检查：验证强拍旋律音是否属于和弦音集合
    · 转换概率检查：基于 chord_transitions.json 统计矩阵验证相邻和弦转换合理性
    · 仅记录异常，不修改和弦数据
    · 输出：覆盖通过率、转换通过率、异常详情列表

  - harmonize-pipeline.ts：端到端和声分析管线
    · 完整流程：MusicXML 解析 → 分段调性分析 → 特征提取 → 分块处理（每 4 小节）
    · 每块：RAG 检索（对应的 2 小节查询）→ LLM 生成 → 难度过滤 → 写入 Score
    · 上下文传递：previousChords 保证分块间和弦连贯性
    · 转调支持：将检测到的转调点写入 Score 的 keyChange 字段
    · 后处理验证：和弦音覆盖 + 转换概率检查
    · 统计信息：总小节数、API 调用次数、处理耗时、分块耗时明细

3.6 输出转换器（src/converter/）

  - ir-to-musicxml.ts：和弦注入 MusicXML
    · 将 Score 中的 ChordSymbol 转为 MusicXML <harmony> 元素
    · 使用正则匹配在原始 XML 的对应小节、对应拍位置的 <note> 前插入
    · 自动检测缩进风格和换行符风格
    · ChordQuality → MusicXML <kind> 映射（11 种）

  - ir-to-abc.ts：ABC Notation 导出
    · 将 Score 转为 ABC Notation 格式文本
    · 包含头部字段：标题、拍号、调号
    · 音符转换：音名 + 八度标记 + 时值后缀
    · 附点音符：精确分数时值计算
    · 和弦标注：双引号括起的和弦符号
    · 休止符支持

  - score-to-render.ts：Verovio 渲染导出
    · MusicXML → SVG（逐页渲染，使用 verovio 引擎）
    · MusicXML → PNG（sharp 转换，多页垂直拼接）
    · MusicXML → PDF（pdfkit 输出，每页对应 PDF 一页，A4 等比缩放）

3.7 Web 前端（web/）

  - index.html：单页应用，约 1400 行，无框架依赖
    · 中国风设计：宋体字族、古纸色调、装饰性音符分隔线
    · 控制栏：上传按钮、加载示例、难度选择（三级）、视图切换、ABC 导出
    · 文件上传：
      - 支持 MusicXML 文件（.xml, .musicxml）→ 直接发送到 /api/harmonize
      - 支持乐谱图片/PDF（.png, .jpg, .pdf 等）→ 发送到 /api/omr/harmonize
      - 支持多页 PDF/图片上传 → /api/omr/harmonize-multi
      - 自动根据文件扩展名路由到对应 API
    · 五线谱渲染：OpenSheetMusicDisplay (OSMD) 1.8.6，支持完整钢琴大谱表
    · 简谱渲染：Canvas 自绘
      - 数字音符（1-7）、八度点（上下）、时值下划线
      - 附点、变音记号、休止符
      - 和弦标注显示在音符上方
      - 自动分行排版
    · 分析面板：调性、置信度、小节数、API 调用次数、处理耗时、和弦进行
    · 验证结果显示：和弦音覆盖通过率、转换概率通过率、异常列表
    · ABC 导出弹窗：显示 ABC 文本 + 复制到剪贴板
    · 内置示例：小星星（Twinkle Twinkle Little Star）12 小节 MusicXML
    · 加载状态：全屏遮罩 + 旋转动画，OMR 和分析显示不同提示文本
    · 错误处理：顶部错误横幅，网络错误/服务器错误友好提示
    · 响应式布局：适配移动端

3.8 后端服务器（server/）

  - index.ts：Express 服务器
    · 端口：4000（可通过 PORT 环境变量配置）
    · 静态文件服务：web/ 目录
    · API 端点：
      - POST /api/harmonize：MusicXML 文本 → 和声分析结果
        输入：{ musicxml: string, difficulty: string }
        输出：{ score, musicxml（带和弦）, analysis, validation, difficultyFilter }
      - POST /api/export/abc：Score → ABC Notation
        输入：{ score: Score }
        输出：{ abc: string }
      - POST /api/omr：乐谱图片/PDF → MusicXML（仅 OMR 识别）
        输入：multipart/form-data（file 字段）
        输出：{ musicxml: string }
      - POST /api/omr/harmonize：乐谱图片/PDF → OMR → 和声分析（一站式）
        输入：multipart/form-data（file 字段）+ difficulty 参数
        输出：与 /api/harmonize 相同
      - POST /api/omr/batch：多页 OMR 识别
        输入：multipart/form-data（files 字段，多文件）
        输出：{ pages: [{ pageIndex, musicxml, filename }] }
      - POST /api/omr/harmonize-multi：多页 OMR → 合并 → 和声分析
        输入：multipart/form-data（files 字段）+ difficulty 参数
        输出：与 /api/harmonize 相同（合并后的完整乐谱）
    · 文件上传：multer 内存存储，限制 20MB
    · 环境变量：DASHSCOPE_API_KEY（必需）、AUDIVERIS_PATH（可选）、PORT（可选）


四、数据资产
────────────────────────────────────────────────────────────────────────────────

  在 git 中的数据文件：
  - data/hooktheory_phrases.json（26MB）：Hooktheory 数据集预处理后的片段数据
    · 来源：Hooktheory 数据库（26,175 首歌曲）
    · 内容：歌曲 ID、艺术家、歌名、调式、和弦序列（RNA）、旋律音程序列
    · 有效片段：125,837 条（过滤掉无和弦或无旋律的条目）

  - data/hooktheory_rna.json（29MB）：Hooktheory 原始数据转 RNA 格式

  - data/chord_transitions.json（0.4MB）：和弦转移概率矩阵（从数据集统计）

  不在 git 中的大文件（需本地生成）：
  - data/phrase_embeddings.bin（492MB）：预计算的 embedding 向量（Float32LE 二进制）
    · 125,837 条 × 1024 维 × 4 字节 = 492MB
    · 使用 text-embedding-v4 模型生成
    · 加载方式：readFileSync → Float32Array 直接映射，加载速度快

  - data/phrase_meta.json（20MB）：与 bin 文件配套的片段元数据
    · 与 hooktheory_phrases.json 内容相同，但只包含有效片段
    · 索引与 phrase_embeddings.bin 一一对应

  - data/phrase_embeddings.json（2.7GB）：原始 embedding JSON 格式（已弃用）
    · 因 V8 字符串长度限制无法用 JSON.parse 加载
    · 已被二进制格式替代，保留作为备份

  生成方式：
    npx tsx scripts/precompute-embeddings.ts   # 生成 phrase_embeddings.json
    npx tsx scripts/convert-embeddings-to-bin.ts  # 转换为 bin + meta 格式


五、脚本工具（scripts/）
────────────────────────────────────────────────────────────────────────────────

  - preprocess-hooktheory.py：Hooktheory 原始数据预处理
    · 将 pitch_class 级别的和弦转为罗马数字级数（RNA）
    · 按 2 小节切片，提取旋律音程序列

  - precompute-embeddings.ts / .py：批量预计算片段 embedding
    · 调用 DashScope text-embedding-v4 API
    · 输出 phrase_embeddings.json 缓存文件

  - convert-embeddings-to-bin.ts：将 JSON embedding 转换为二进制格式
    · 使用 stream-json 流式解析 2.7GB JSON（避免内存溢出）
    · 输出 phrase_meta.json + phrase_embeddings.bin
    · 支持断点续传（已存在的文件自动跳过）

  - inspect-hooktheory.py：Hooktheory 数据检查/调试工具

  - test-omr.ts：OMR 引擎 CLI 测试脚本
  - test-inject2.ts：和弦注入测试脚本
  - test-pipeline.ts：端到端管线测试脚本

十、性能特征
────────────────────────────────────────────────────────────────────────────────

  使用二进制预计算 embedding 时（推荐）：
  - 数据加载：~2-3 秒（492MB bin + 20MB meta）
  - 每次 RAG 检索：~1-2 秒（1 次 embedding API 调用 + 本地余弦相似度）
  - 32 小节乐谱端到端：~25-30 秒（8 个 chunk × ~3s）
  - API 调用：~24 次（16 次 RAG embedding + 8 次 LLM）

  无预计算 embedding 时（回退模式，极慢）：
  - 每次 RAG 检索：~50 秒（200 个候选 × 实时 embedding）
  - 32 小节乐谱端到端：~400+ 秒
  - API 调用：~340+ 次


十一、当前状态与已知限制
────────────────────────────────────────────────────────────────────────────────

  已完成功能：
  - 完整的 MusicXML 解析 → 调性分析 → RAG+LLM 和声分析 → 输出渲染流程
  - OMR 乐谱图片识别（Audiveris）集成，前后端打通
  - 多页 OMR 识别 + MusicXML 合并
  - 五线谱（OSMD）和简谱（Canvas）双视图渲染
  - ABC Notation 导出
  - Verovio PNG/PDF 导出
  - 三级难度和弦配置 + 白名单过滤
  - 后处理验证（和弦音覆盖 + 转换概率）
  - 分段调性分析 + 转调检测
  - 二进制 embedding 存储（解决 2.7GB JSON 加载失败问题）
  - Web 前端完整交互
  - GitHub 仓库：https://github.com/evenstar123/Melochrod

  已知限制：
  - 多声部处理：解析时只取第一个 part（右手/主旋律），左手伴奏不参与分析
  - 简谱渲染和 ABC 导出为单声部输出
  - OMR 依赖本地安装的 Audiveris，不支持云端部署
  - 和弦编辑功能尚未实现（用户无法手动修正生成的和弦）
  - 无用户认证和多用户支持
